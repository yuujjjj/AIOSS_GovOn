# 모델 카드

이 문서는 GovOn MVP가 전제하는 **모델 스택**을 설명한다. 현재 기준은 "하나의 base model + 필요 시 civil-response adapter" 구조이며, 공문서 adapter와 분류 모델은 MVP 범위에서 제외한다.

## 모델 구조

| 항목 | 내용 |
|------|------|
| 제품 표면 | `govon` CLI 셸 |
| 기본 모델 | EXAONE 계열 추론 베이스 모델 |
| 서빙 엔진 | vLLM |
| 특화 능력 | civil-response LoRA adapter |
| 기본 책임 | 의도 파악, 작업 계획, tool 사용 판단, 응답 합성 |
| adapter 책임 | 민원 답변 작성 단계의 문안 생성 품질 향상 |

## 사용 원칙

### Base model

base model은 항상 살아 있는 기본 두뇌다. 다음 역할을 맡는다.

- 사용자 요청 이해
- 이번 턴의 한 작업 정의
- tool 필요 여부 판단
- 승인 전 설명 문안 생성
- 도구 결과 통합
- 최종 응답 합성

### Civil-response adapter

civil-response adapter는 항상 켜져 있지 않다.

- 사용 조건: 사용자의 의도가 실제 민원 답변 작성 단계로 들어갔을 때
- 승인 정책: 작업 승인 범위 안에 포함
- 사용자 노출 방식: 별도 모델 전환 UI 없이 내부 capability처럼 동작

## MVP 제외 모델 경로

- 공문서 작성 adapter
- 분류 전용 모델
- 웹 전용 응답 모델

## RAG와의 관계

민원 답변 초안은 품질 향상을 위해 내부 검색을 사용할 수 있다. 다만 출처는 기본 출력이 아니며, 사용자가 후속으로 근거를 요청할 때 다시 검색을 수행해 기존 답변 아래에 증강한다.

## 알려진 제약

- adapter attach/swap 방식은 실제 추론 엔진 제약을 고려해 구현해야 한다.
- RAG 검증은 초기에는 샘플 문서 기준으로 수행한다.
- API 출처 포맷은 실제 응답 형태를 확인한 뒤 구체화한다.

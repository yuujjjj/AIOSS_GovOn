# ADR-002: AWQ W4A16g128 양자화 방식 선정

## Status

Accepted

## Context

EXAONE-Deep-7.8B 모델을 QLoRA로 파인튜닝한 후, 공공기관 온프레미스 환경의 제한된 GPU VRAM(16~24GB)에서 추론 서빙을 해야 한다. BF16 전체 정밀도 모델은 약 15.6GB VRAM을 차지하여, 모델 로딩만으로 VRAM이 거의 소진된다. KV 캐시와 배치 처리를 위한 여유 공간을 확보하려면 양자화가 필수적이다.

양자화 방식 선택 시 다음 요구사항을 충족해야 한다.

1. **추론 품질 유지**: 민원 분류 정확도와 답변 생성 품질이 FP16 대비 크게 저하되지 않아야 한다.
2. **추론 속도**: 실시간에 가까운 응답 속도(토큰/초)를 달성해야 한다.
3. **서빙 프레임워크 호환**: vLLM과의 네이티브 통합이 가능해야 한다.
4. **양자화 파이프라인 복잡도**: 팀이 유지보수할 수 있는 수준의 양자화 절차여야 한다.

## 검토 후보

| 후보 | 비트 수 | 커널 최적화 | vLLM 지원 | 모델 크기(추정) | 캘리브레이션 필요 |
|------|---------|------------|-----------|----------------|------------------|
| **AWQ W4A16g128** | W4A16 (그룹 크기 128) | GEMM 커널 최적화 | 네이티브 지원 | ~4.5GB | 도메인 데이터 512샘플 |
| GPTQ | W4A16 | ExLlama v2 커널 | 지원 (Marlin 백엔드) | ~4.5GB | 도메인 데이터 필요 |
| BitsAndBytes INT8 | W8A8 | LLM.int8() | 제한적 지원 | ~8GB | 불필요 |
| BitsAndBytes NF4 | W4A16 | QLoRA 전용 | 미지원 (학습 전용) | ~5GB | 불필요 |
| FP16 (비양자화) | W16A16 | 해당 없음 | 지원 | ~15.6GB | 불필요 |

### 상세 비교

**AWQ W4A16g128**
- Activation-aware Weight Quantization의 약자로, 활성화 분포를 고려하여 중요한 가중치 채널의 양자화 오차를 최소화한다.
- 그룹 크기 128(g128)은 양자화 세밀도와 속도의 균형점이다. 그룹 크기가 작을수록 품질은 올라가지만 커널 오버헤드가 증가한다.
- vLLM이 AWQ 모델을 직접 로드하여 GEMM 커널로 추론하므로, 별도 변환 없이 서빙이 가능하다.
- AutoAWQ 라이브러리(`autoawq>=0.2.0`)로 양자화를 수행하며, 도메인 캘리브레이션 데이터로 민원 학습 데이터 512샘플을 사용한다.

**GPTQ**
- AWQ와 유사한 4비트 양자화 방식이나, 가중치 재구성 오차를 최소화하는 방식으로 접근한다.
- vLLM에서 Marlin 백엔드를 통해 지원되나, AWQ 대비 vLLM 통합 성숙도가 낮고 일부 모델 아키텍처에서 호환 이슈가 보고되었다.
- EXAONE 커스텀 모델 코드와의 조합에서 AWQ보다 검증 사례가 적었다.

**BitsAndBytes INT8/NF4**
- INT8은 양자화 비트 수가 높아 모델 크기 절감 효과가 AWQ 대비 절반 수준이다.
- NF4는 QLoRA 학습 시 사용하는 양자화 방식으로, 추론 서빙에 최적화된 커널이 없다. vLLM에서 공식 지원하지 않아 서빙 프레임워크로 사용할 수 없다.

**FP16 (비양자화)**
- 모델 로딩만으로 ~15.6GB를 사용하여, 16GB GPU에서는 KV 캐시 공간이 부족하고 24GB GPU에서도 동시 처리 용량이 극도로 제한된다.
- 성능 기준선으로만 활용하고 프로덕션 서빙에는 부적합하다.

## Decision

**AWQ W4A16g128** 양자화를 적용한다.

양자화 파이프라인은 다음 2단계로 구성한다.

1. **Stage 1 -- LoRA 병합** (`src/quantization/merge_lora.py`): QLoRA 어댑터를 베이스 모델에 BF16으로 병합한다. 병합 후 파라미터 수가 베이스 모델과 일치하는지 검증하고, 추론 테스트로 무결성을 확인한다.
2. **Stage 2 -- AWQ 양자화** (`src/quantization/quantize_awq.py`): 병합된 BF16 모델에 AutoAWQ로 W4A16g128 양자화를 적용한다. 캘리브레이션 데이터는 민원 학습 데이터에서 512샘플을 추출하여 도메인 특화 양자화 품질을 확보한다.

양자화 설정:
- `w_bit`: 4 (4비트 가중치)
- `q_group_size`: 128 (128개 가중치를 하나의 양자화 그룹으로 묶음)
- `zero_point`: True (비대칭 양자화로 정밀도 향상)
- `version`: GEMM (vLLM 호환 GEMM 커널 사용)

## Consequences

### 긍정적 영향

- 모델 크기가 ~15.6GB에서 ~4.5GB로 약 3.5배 축소되어, 16GB GPU에서 KV 캐시 공간을 충분히 확보할 수 있다.
- vLLM이 AWQ 모델을 네이티브로 로드하므로, 양자화 모델을 별도 변환 없이 곧바로 서빙에 사용할 수 있다.
- 도메인 캘리브레이션 데이터를 사용하여, 범용 캘리브레이션 대비 민원 텍스트에 대한 양자화 오차를 줄인다.
- WandB 연동으로 양자화 전후 모델 크기, 압축률, 소요 시간 등을 추적한다.

### 부정적 영향

- `autoawq>=0.2.0` 의존성이 추가된다. AutoAWQ는 CUDA 빌드가 필요하며, CPU 환경에서는 양자화 수행이 불가하다.
- 캘리브레이션 데이터셋(512샘플)이 필요하므로, 학습 데이터 파이프라인과 양자화 파이프라인 간 의존성이 생긴다.
- 4비트 양자화로 인해 FP16 대비 소폭의 품질 저하가 불가피하다. 특히 숫자 추론이나 정밀한 법령 조항 인용에서 오차가 발생할 수 있다.
- EXAONE 커스텀 모델 코드(`modeling_exaone.py`, `configuration_exaone.py`)를 양자화된 모델 디렉토리에 함께 복사해야 하는 추가 관리 포인트가 존재한다.

### 향후 고려사항

- AWQ 양자화 품질을 주기적으로 평가 파이프라인(`src/evaluation/`)으로 모니터링하고, FP16 대비 성능 저하가 허용 범위를 초과하면 그룹 크기를 64로 줄이거나 GPTQ로 전환을 검토한다.
- AutoAWQ의 메이저 버전 업데이트 시 양자화 결과물의 호환성을 확인해야 한다.
- GPU VRAM이 충분한 환경(48GB 이상)으로 전환될 경우, FP16 서빙으로 전환하여 양자화 없이 최대 품질을 확보하는 방안을 재검토한다.

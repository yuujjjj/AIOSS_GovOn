# ADR-004: 로컬 RAG 검색 엔진으로 FAISS 유지

## Status

Accepted

## Context

GovOn MVP는 민원 답변 품질을 높이기 위해 로컬 RAG를 유지한다. 다만 RAG는 제품의 본체가 아니라 `govon` 셸의 내부 capability 중 하나이며, 초기 단계에서는 실제 운영 문서가 아닌 샘플 문서로 기능 검증을 수행한다. 검색 계층은 다음 조건을 만족해야 한다.

1. **로컬 파일 기반 운용**: 지정 폴더에 적재된 문서를 인덱싱하고 검색할 수 있어야 한다.
2. **다양한 문서 형식**: `pdf`, `hwp`, `docx`, `txt`, `html`을 순차 지원해야 한다.
3. **로컬 데몬 결합성**: FastAPI daemon 안에서 직접 호출할 수 있어야 한다.
4. **단순한 운영 모델**: MVP 단계에서 별도 벡터 DB 서버를 추가하지 않아야 한다.
5. **출처 증강 적합성**: 후속 근거 요청 시 `파일경로 + 페이지` 기준으로 근거를 덧붙일 수 있어야 한다.

## 검토 후보

| 후보 | 장점 | 단점 |
|------|------|------|
| **FAISS** | 프로세스 내 임베디드 사용, 파일 기반 인덱스, 운영 단순 | 메타데이터 필터링이 제한적, 삭제/수정이 불편 |
| Chroma | 임베디드 사용 가능, 사용 편의성 높음 | 라이브러리 내부 복잡도가 높고 장기 운영 경계가 애매함 |
| Qdrant | 메타데이터 필터링/운영 기능 풍부 | 별도 서버 운영 필요, MVP에 과함 |
| pgvector | DB에 통합 가능 | 검색 성능과 인덱스 관리 측면에서 전용 엔진보다 불리 |

## Decision

GovOn의 로컬 RAG 검색 엔진은 **FAISS + BM25 하이브리드 구조**를 유지한다.

다만 기존의 "대규모 폐쇄망 인덱스" 전제보다, MVP에서는 다음처럼 더 단순하게 정의한다.

- 지정 폴더의 문서를 로컬 daemon이 인덱싱한다.
- 샘플 문서로도 전체 검색 경로를 검증할 수 있어야 한다.
- 검색 결과는 민원 답변 생성 품질 개선용 내부 컨텍스트로 우선 사용한다.
- 사용자가 후속으로 근거를 요구했을 때만 `파일경로 + 페이지` 기준 출처를 노출한다.

### 검색 계층 역할

- Dense search: FAISS
- Sparse search: BM25
- Fusion: Hybrid search
- Provenance: 문서 경로, 페이지, 발췌 텍스트

## Consequences

### 긍정적 영향

- 별도 검색 서버 없이 로컬 daemon 내부에서 RAG를 처리할 수 있다.
- MVP에서 샘플 문서만으로도 ingestion, indexing, retrieval, evidence augmentation 흐름을 모두 검증할 수 있다.
- 벡터 DB 운영 부담 없이 빠르게 기능을 구현할 수 있다.

### 부정적 영향

- 문서 수정/삭제가 잦은 환경에서는 인덱스 재빌드 관리가 필요하다.
- `pdf`, `hwp`처럼 페이지/텍스트 추출이 불완전한 포맷은 provenance 품질 편차가 생길 수 있다.
- 메타데이터 기반 복잡한 필터링은 애플리케이션 레벨에서 처리해야 한다.

### 향후 고려사항

- 운영 문서 확보 후 샘플 인덱스와 실운영 인덱스의 분리 전략이 필요하다.
- 페이지 단위 provenance 정확도가 충분하지 않으면 문서 형식별 추출기 개선이 필요하다.
- 메타데이터 필터링 요구가 커지면 Qdrant 또는 SQLite FTS 조합을 재검토할 수 있다.

## 1. 개요

### 1.1 문제 정의 (요약)

공공기관 민원 처리는 **키워드 기반 오분류**와 **비효율적인 수작업 프로세스** 두 가지 핵심 문제를 안고 있다. 민원인은 폭증하는 반면 담당 공무원은 1인이 하루 20~30건을 수작업으로 처리해야 하며, 이 과정에서 동음이의어·유사 표현으로 인한 오분류가 반복적으로 발생한다.

### 1.2 주 고객

> **본 시스템의 주 고객은 민원인이 아니라, 민원 처리·분류를 담당하는 공무원이다.** 민원인은 기존 채널(온라인·오프라인)을 통해 민원을 접수하며, AI 시스템과 직접 상호작용하지 않는다. 본 시스템은 접수된 민원을 **공무원이 더 빠르고 정확하게 처리**할 수 있도록 지원하는 내부 업무 도구이다.

### 1.3 제안 솔루션

폐쇄망(온프레미스) 환경에서 구동되는 한국어 특화 LLM(**EXAONE-Deep-7.8B**)을 QLoRA로 파인튜닝하여, 민원 자동 분류 및 표준 답변 초안 생성을 지원하는 On-Device AI 시스템을 개발한다. AI Hub 공개 데이터와 크롤링 데이터를 결합하여 학습하고, FAISS 기반 벡터 검색(RAG)으로 유사 사례를 즉시 참조할 수 있도록 한다.

### 1.4 핵심 비즈니스 영향

| 영역 | 기대 효과 |
|------|----------|
| 업무 효율 | 민원 처리 시간 평균 60% 이상 단축 |
| 분류 정확도 | 기존 키워드 방식 대비 30%p 향상 (목표 85% 이상) |
| 보안 규정 준수 | 망분리 의무, 개인정보보호법, 국정원 보안 지침을 100% 준수하는 폐쇄망 처리 |
| 운영비 절감 | 외부 클라우드 API 종속 없이 온프레미스 단독 운영 → 연간 유지비 90% 절감 |

> **관점**: 공공기관이 클라우드 AI를 도입하지 않는 것은 예산이 부족해서가 아니라, 망분리 규정과 보안 심의 절차상 **허가받기 어려운 구조**이기 때문이다. 본 시스템은 그 제약 안에서 비용 절감 효과까지 실현한다.

## 2. 배경 및 문제 정의

### 2.1 핵심 문제 (우선순위 순)

> **교수님 피드백 반영**: 문제점 1은 간략히, 문제점 2·3이 핵심, 문제점 4 제거

#### 문제점 1 (참고) : 공공기관 폐쇄망 환경의 구조적 제약

공공기관 민원 데이터에는 주민등록번호·주소·연락처 등 민감 정보가 포함되어 있어, 외부 클라우드 AI(ChatGPT, Claude API)는 **보안 규정 자체**로 사용이 차단된다. 이는 예산의 문제가 아니라, 아래와 같은 제도적·행정적 장벽에 기인한다.

**공공기관이 외부 클라우드 AI를 도입할 수 없는 이유**:

| 규정 / 절차 | 내용 |
|------------|------|
| **망분리 의무** | 국가정보원 「전자정부 SW 개발·운영 자문기준」 및 「정보보안 기본지침」에 따라 공공기관 내부망은 외부 인터넷과 물리적·논리적으로 분리되어야 함 |
| **개인정보보호법** | 제17조 · 제18조: 개인정보를 제3자(클라우드 API 서버)에 제공 시 정보주체 동의 또는 법적 근거 필요 → 민원 데이터의 외부 전송은 원칙적으로 불가 |
| **전자정부법** | 제56조: 행정정보 보호를 위한 보안 조치 의무화 — 외부망 경유 AI 서비스는 보안적합성 심의 대상 |
| **보안 심의 절차** | 국가사이버안전센터(NCSC) 보안적합성 검토 → 클라우드 AI 서비스는 승인까지 6~12개월 소요, 사실상 도입 불가 수준 |
| **개인정보 영향평가(PIA)** | 공공기관이 개인정보 처리 시스템 도입 시 PIA 수행 의무 (「개인정보보호법」 제33조) — 외부 API 연동은 평가 범위가 크게 확대되어 도입 장벽 상승 |

따라서 폐쇄망 온프레미스 처리는 **보안 규정 준수를 위한 필수 요건**이며, 이 제약 안에서 On-Device AI를 구동하는 것이 본 프로젝트의 출발점이다.

---

#### 문제점 2 (핵심) : 키워드 기반 분류의 **동음이의어·오분류** 한계

기존 시스템은 단순 키워드 매칭 방식으로 동작하므로 다음과 같은 오분류가 반복된다.

```
민원 예시: "도로 파손"  →  키워드: "파손"  →  [시설파손] 카테고리로 오분류
올바른 분류: [도로/교통] 카테고리
```

| 오분류 유형 | 예시 |
|------------|------|
| **동음이의어** | "배수" → 배수관 민원인데 산술 관련으로 분류 |
| **유사 표현** | "도로 파손" vs "보도 파손" 구분 불가 |
| **신조어·지역 방언** | 지역 특유 표현에 대응 불가 |
| **맥락 의존적 의미** | "소음 민원"이 공사·교통·주거 중 어느 카테고리인지 문맥 없이 판단 불가 |

현재 키워드 방식의 분류 정확도는 약 55% 수준으로 추정되며, 오분류된 민원은 담당 부서가 바뀌어 처리 지연 및 이중 업무가 발생한다.

---

#### 문제점 3 (핵심) : 비효율적 수작업 프로세스

**민원인은 폭증하는데, 담당 공무원은 1인** 구조에서 아래와 같은 시간 손실이 발생한다.

```
민원 1건당 처리 흐름 (현재):
민원 접수 → 읽기(5분) → 분류(5분) → 유사사례 수동 검색(10분) → 답변 작성(10-15분)
= 합계 약 30~35분 / 건

하루 25건 처리 시 → 약 13시간 소요 (실질적으로 초과근무 발생)
```

- 민원 1건당 평균 처리 시간: **15~30분** (답변 작성 포함)
- 연간 민원 처리 건수: 약 **1,500만 건 이상** (행정안전부 민원포털 기준)
- 국내 지자체 수: **243개** (광역 17개, 기초 226개)
- 담당자의 경험·숙련도에 따라 답변 품질 편차가 크며, 신규 공무원의 경우 유사 사례를 찾는 데 더 많은 시간이 소요된다.

### 2.2 폐쇄망 환경에서 RAG가 필요한 이유

폐쇄망 환경에서 LLM 단독 운영 시 발생하는 치명적 한계가 있으며, 이를 RAG(Retrieval-Augmented Generation)로 보완한다.

#### LLM 단독 운영의 한계

```
[폐쇄망 LLM 단독 사용 시 문제]

LLM 학습 데이터 고정 (지식 컷오프 존재)
    │
    ├─ 최신 조례 개정 반영 불가
    ├─ 새로운 민원 유형 대응 어려움
    └─ 기관별 표준 답변 양식 업데이트 반영 불가

외부 검색엔진 연결 불가 (폐쇄망 특성상)
    │
    └─ 실시간 법령·지침 조회 불가능

모델 재학습 비용 부담
    │
    └─ 신규 데이터 반영 시마다 Fine-tuning 재수행 필요
       → GPU 자원·인력·시간 소모
```

#### RAG 도입으로 해결하는 문제

| 문제 | RAG 해결 방법 |
|------|--------------|
| **지식 고정 (Knowledge Cutoff)** | 최신 민원 사례·조례·지침을 벡터 DB에 주기적으로 추가 → 모델 재학습 없이 최신 정보 반영 |
| **기관별 맞춤 답변 부재** | 해당 기관의 실제 답변 이력을 벡터 DB로 구축 → 기관 특화 표현·서식 자동 참조 |
| **환각(Hallucination) 위험** | 검색된 실제 사례를 근거로 답변 생성 → 사실 기반 답변 신뢰성 향상 |
| **신규 공무원 적응** | 유사 사례 상위 5건 제시 → 경험 부족 공무원도 즉시 참조 가능 |

> **핵심 논거**: 폐쇄망에서는 외부 검색도 불가능하고 모델도 업데이트하기 어렵기 때문에, **내부 데이터 기반 RAG**가 실시간 데이터 활용의 유일한 현실적 대안이다.

### 2.3 시장 및 환경 분석

> **교수님 피드백 반영**: 담당 공무원(1인) vs. 민원인(다수) 구도 명확화, 팩트체크 근거 포함

│   - 분류                 │
│   - 사례 검색            │
│   - 답변 작성            │
└─────────────────────────┘
```

| 민원 1건당 평균 처리 시간 | 15~30분 | 현장 인터뷰 추정치 |

## 4. 사용자 분석

> **주 고객: 민원 처리·분류 담당 공무원** — 본 시스템은 민원인을 대상으로 하는 서비스가 아니며, 공무원의 민원 업무 효율화를 위한 내부 도구이다. 민원인은 기존 접수 채널을 그대로 사용하며 AI 시스템과 직접 접촉하지 않는다.

### 4.1 페르소나

> **교수님 피드백 반영**: 페르소나 1 유지·강화, 페르소나 2 보강

#### 페르소나 1: 민원 담당 공무원 (주 고객)

```
┌─────────────────────────────────────────────────────────┐
│  김민원 (35세, 행정직 6급)                               │
│  "하루 종일 민원 답변만 쓰다 보면 퇴근 시간이 없어요"      │
├─────────────────────────────────────────────────────────┤
│  역할     │ 지자체 민원 담당 공무원                       │
│  IT 수준  │ 중급 (인터넷·행정 시스템 사용 능숙)           │
│  하루 업무│ 민원 20~30건 처리 + 유선 응대 + 내부 보고     │
├─────────────────────────────────────────────────────────┤
│  목표                                                   │

## 5. 기술 스택 및 아키텍처

> **교수님 피드백 반영**: LLM 외 균형 있는 기술 요소 표현, 아키텍처 다이어그램 중심

### 5.1 기술 스택

> **LLM이 전부가 아닌, 데이터 파이프라인·벡터 검색·서빙·인프라가 함께 구성된 시스템**

#### 모델 경량화(양자화)가 필요한 이유

> **관점**: 모델 경량화는 "예산이 부족해서"가 아니라, **폐쇄망 온프레미스 환경의 운영 효율화 및 비용 최적화**를 위한 전략적 선택이다.

공공기관은 외부 클라우드 GPU를 임의로 사용할 수 없다. 내부 서버에 탑재된 GPU 사양 범위 안에서 LLM을 운영해야 하며, 서버 구매·전력·유지보수 비용 모두 기관 예산 심의를 거쳐야 한다. 이 환경에서 모델 경량화는 다음의 효과를 달성한다.

| 경량화 목표 | 설명 |
|------------|------|
| **하드웨어 요건 완화** | BF16 원본(~20GB VRAM) → AWQ 4-bit(~4.2GB VRAM) → RTX 3060급 GPU에서 운영 가능 |
| **서버 비용 절감** | A100(연간 수천만 원 수준 GPU 서버) 없이 중저가 GPU 서버로 동일 성능 달성 |
| **전력·냉각 비용 절감** | VRAM 사용량 감소 → 전력 소비 감소 → 데이터센터 운영비 절감 |
| **다기관 확장 용이** | 경량 모델은 보급형 서버에도 설치 가능 → 243개 지자체 전체 보급 현실화 |

실측 결과: AWQ 4-bit 양자화 적용 시 모델 크기 15.6GB → 4.94GB (-68.3%), 분류 정확도는 90.0%로 목표(85%) 초과 달성.

> **⚠ 모델 현황 (2026-03-19)**: 기존 HuggingFace의 AWQ 모델(`umyunsang/civil-complaint-exaone-awq`)은 잘못 학습된 LoRA v1 기반 병합 모델로 생성되어 **폐기** 처리되었다. 현재 재학습된 LoRA v2(`umyunsang/GovOn-EXAONE-LoRA-v2`)를 기반으로 AWQ 양자화 모델을 재생성할 예정이다.

## 7. 수락 기준

### 7.1 핵심 기능 수락 기준

#### AC-001: 민원 자동 분류 (핵심)
**Given** 민원 본문 텍스트가 입력되었을 때
**When** 시스템이 분류를 실행하면
**Then**
- 테스트 데이터셋 1,000건 기준 분류 정확도 **≥ 85%**
- 동음이의어 테스트 케이스 포함 검증 완료
- 분류 결과가 0.5초 이내 반환

#### AC-002: 답변 초안 생성
**Given** 민원 본문이 입력되었을 때
**When** "답변 생성" 버튼을 클릭하면
**Then**
- p95 < 3초 이내 답변 초안 생성
- 생성된 답변이 표준 서식(인사말·본문·담당부서·연락처)을 포함
- 답변 길이 200~500자 범위

#### AC-003: 유사 사례 검색
**Given** 민원 본문이 입력되었을 때
**When** 벡터 검색을 실행하면
**Then**
- 상위 5개 유사 민원-답변 쌍 반환
- 검색 시간 p95 < 1초 (FAISS 벡터 DB 레이턴시 기준)
- Recall@5 ≥ 80%

#### AC-004: Docker 폐쇄망 배포
**Given** Docker + nvidia-container-toolkit이 설치된 서버가 준비되었을 때
**When** `docker-compose up -d` 명령을 실행하면
**Then**
- 인터넷 연결 없이 전체 시스템 정상 시작
- `/health` 엔드포인트 200 응답 반환
- 웹 UI Port 3000에서 접근 가능

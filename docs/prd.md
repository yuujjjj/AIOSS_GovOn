## 1. 개요

### 1.1 문제 정의 (요약)

공공기관 민원 처리는 **키워드 기반 오분류**와 **비효율적인 수작업 프로세스** 두 가지 핵심 문제를 안고 있다. 담당 공무원이 민원을 수작업으로 처리하는 과정에서 동음이의어·유사 표현으로 인한 오분류가 반복적으로 발생한다.

### 1.2 주 고객

> **본 시스템의 주 고객은 민원인이 아니라, 민원 처리·분류를 담당하는 공무원이다.** 민원인은 기존 채널(온라인·오프라인)을 통해 민원을 접수하며, AI 시스템과 직접 상호작용하지 않는다. 본 시스템은 접수된 민원을 **공무원이 더 빠르고 정확하게 처리**할 수 있도록 지원하는 내부 업무 도구이다.

### 1.3 제안 솔루션

폐쇄망(온프레미스) 환경에서 구동되는 한국어 특화 LLM(**EXAONE-Deep-7.8B**)을 QLoRA로 파인튜닝하여, 민원 자동 분류 및 표준 답변 초안 생성을 지원하는 폐쇄망 로컬 LLM 시스템을 개발한다. AI Hub 공개 데이터와 크롤링 데이터를 결합하여 학습하고, FAISS 기반 벡터 검색(RAG)으로 유사 사례를 즉시 참조할 수 있도록 한다.

### 1.4 핵심 비즈니스 영향

| 영역 | 기대 효과 |
|------|----------|
| 업무 효율 | AI 자동 분류·답변 초안 생성으로 민원 처리 시간 단축 |
| 분류 정확도 | LLM 기반 문맥 분류로 키워드 방식 대비 정확도 향상 (목표 85% 이상) |
| 보안 규정 준수 | 망분리 의무, 개인정보보호법, 국정원 보안 지침을 100% 준수하는 폐쇄망 처리 |
| 운영비 절감 | 외부 클라우드 API 종속 없이 온프레미스 단독 운영 |

> **관점**: 공공기관이 클라우드 AI를 도입하지 않는 것은 예산이 부족해서가 아니라, 망분리 규정과 보안 심의 절차상 **허가받기 어려운 구조**이기 때문이다. 본 시스템은 그 제약 안에서 비용 절감 효과까지 실현한다.

## 2. 배경 및 문제 정의

### 2.1 핵심 문제 (우선순위 순)

> **교수님 피드백 반영**: 문제점 1은 간략히, 문제점 2·3이 핵심, 문제점 4 제거

#### 문제점 1 (참고) : 공공기관 폐쇄망 환경의 구조적 제약

공공기관 민원 데이터에는 주민등록번호·주소·연락처 등 민감 정보가 포함되어 있어, 외부 클라우드 AI(ChatGPT, Claude API)는 **보안 규정 자체**로 사용이 차단된다. 이는 예산의 문제가 아니라, 아래와 같은 제도적·행정적 장벽에 기인한다.

**공공기관이 외부 클라우드 AI를 도입할 수 없는 이유**:

| 규정 / 절차 | 내용 |
|------------|------|
| **망분리 의무** | 「국가정보보안기본지침」 제40·41조에 따라 공공기관 내부망은 외부 인터넷과 물리적·논리적으로 분리되어야 함 ([NCSC 공식 지침](https://www.ncsc.go.kr:4018/main/cop/bbs/selectBoardArticle.do?bbsId=InstructionGuide_main&nttId=18588&pageIndex=1)) |
| **개인정보보호법** | 제17조(개인정보의 제공) · 제18조(목적 외 이용·제공 제한): 정보주체 동의 없이 제3자(클라우드 API 서버)에 민원 데이터를 전송하는 행위는 원칙적으로 위반 소지 있음 ([국가법령정보센터 제17조](https://www.law.go.kr/lsLawLinkInfo.do?lsJoLnkSeq=900078379&chrClsCd=010202), [제18조](https://www.law.go.kr/%EB%B2%95%EB%A0%B9/%EA%B0%9C%EC%9D%B8%EC%A0%95%EB%B3%B4%20%EB%B3%B4%ED%98%B8%EB%B2%95/%EC%A0%9C18%EC%A1%B0)) |
| **전자정부법** | 제56조 제1·2·3항: 행정기관의 정보통신망·행정정보 보안대책 수립 의무 및 국가정보원장 확인 보안조치 사용 의무 ([법령 원문](https://lawnb.com/Info/ContentView?sid=L000009199_56_20220705)) |
| **개인정보 영향평가(PIA)** | 공공기관이 개인정보 처리 시스템 도입 시 PIA 수행 의무 (「개인정보보호법」 제33조) — 외부 API 연동은 평가 범위가 크게 확대되어 도입 장벽 상승 |

따라서 폐쇄망 온프레미스 처리는 **보안 규정 준수를 위한 필수 요건**이며, 이 제약 안에서 로컬 LLM을 구동하는 것이 본 프로젝트의 출발점이다.

#### 문제점 2 (핵심) : 키워드 기반 분류의 **동음이의어·오분류** 한계

기존 시스템은 단순 키워드 매칭 방식으로 동작하므로 다음과 같은 오분류가 반복된다.

```
민원 예시: "도로 파손"  →  키워드: "파손"  →  [시설파손] 카테고리로 오분류
올바른 분류: [도로/교통] 카테고리
```

| 오분류 유형 | 예시 |
|------------|------|
| **동음이의어** | "배수" → 배수관 민원인데 산술 관련으로 분류 |
| **유사 표현** | "도로 파손" vs "보도 파손" 구분 불가 |
| **신조어·지역 방언** | 지역 특유 표현에 대응 불가 |
| **맥락 의존적 의미** | "소음 민원"이 공사·교통·주거 중 어느 카테고리인지 문맥 없이 판단 불가 |

오분류된 민원은 담당 부서가 바뀌어 처리 지연 및 이중 업무가 발생한다.

#### 문제점 3 (핵심) : 비효율적 수작업 프로세스

**담당 공무원 1인** 구조에서 민원 접수 → 분류 → 유사사례 검색 → 답변 작성의 전 과정을 수작업으로 처리해야 하며, 아래와 같은 비효율이 발생한다.

```
민원 1건당 처리 흐름 (현재):
민원 접수 → 읽기·내용 파악 → 분류 → 유사사례 수동 검색 → 답변 작성 → 검토·발송
```

- 국내 지자체 수: **243개** (광역 17개, 기초 226개) ([행정안전부 공식 문서](https://ko.wikipedia.org/wiki/%EB%8C%80%ED%95%9C%EB%AF%BC%EA%B5%AD%EC%9D%98_%EA%B8%B0%EC%B4%88%EC%9E%90%EC%B9%98%EB%8B%A8%EC%B2%B4_%EB%AA%A9%EB%A1%9D))
- 담당자의 경험·숙련도에 따라 답변 품질 편차가 크며, 신규 공무원의 경우 유사 사례를 찾는 데 더 많은 시간이 소요된다.

### 2.2 폐쇄망 환경에서 RAG가 필요한 이유

폐쇄망 환경에서 LLM 단독 운영 시 발생하는 치명적 한계가 있으며, 이를 RAG(Retrieval-Augmented Generation)로 보완한다.

#### LLM 단독 운영의 한계

```
[폐쇄망 LLM 단독 사용 시 문제]

LLM 학습 데이터 고정 (지식 컷오프 존재)
    │
    ├─ 최신 조례 개정 반영 불가
    ├─ 새로운 민원 유형 대응 어려움
    └─ 기관별 표준 답변 양식 업데이트 반영 불가

외부 검색엔진 연결 불가 (폐쇄망 특성상)
    │
    └─ 실시간 법령·지침 조회 불가능

모델 재학습 비용 부담
    │
    └─ 신규 데이터 반영 시마다 Fine-tuning 재수행 필요
       → GPU 자원·인력·시간 소모
```

#### RAG 도입으로 해결하는 문제

| 문제 | RAG 해결 방법 |
|------|--------------|
| **지식 고정 (Knowledge Cutoff)** | 최신 민원 사례·조례·지침을 벡터 DB에 주기적으로 추가 → 모델 재학습 없이 최신 정보 반영 |
| **기관별 맞춤 답변 부재** | 해당 기관의 실제 답변 이력을 벡터 DB로 구축 → 기관 특화 표현·서식 자동 참조 |
| **환각(Hallucination) 위험** | 검색된 실제 사례를 근거로 답변 생성 → 사실 기반 답변 신뢰성 향상 |
| **신규 공무원 적응** | 유사 사례 상위 5건 제시 → 경험 부족 공무원도 즉시 참조 가능 |

> **핵심 논거**: 폐쇄망에서는 외부 검색도 불가능하고 모델도 업데이트하기 어렵기 때문에, **내부 데이터 기반 RAG**가 실시간 데이터 활용의 유일한 현실적 대안이다.

## 4. 사용자 분석

> **주 고객: 민원 처리·분류 담당 공무원** — 본 시스템은 민원인을 대상으로 하는 서비스가 아니며, 공무원의 민원 업무 효율화를 위한 내부 도구이다. 민원인은 기존 접수 채널을 그대로 사용하며 AI 시스템과 직접 접촉하지 않는다.

## 5. 기술 스택 및 아키텍처

> **교수님 피드백 반영**: LLM 외 균형 있는 기술 요소 표현, 아키텍처 다이어그램 중심

### 5.1 기술 스택

> **LLM이 전부가 아닌, 데이터 파이프라인·벡터 검색·서빙·인프라가 함께 구성된 시스템**

#### 모델 경량화(양자화)가 필요한 이유

> **관점**: 모델 경량화는 "예산이 부족해서"가 아니라, **폐쇄망 온프레미스 환경의 운영 효율화 및 비용 최적화**를 위한 전략적 선택이다.

공공기관은 외부 클라우드 GPU를 임의로 사용할 수 없다. 내부 서버에 탑재된 GPU 사양 범위 안에서 LLM을 운영해야 하며, 서버 구매·전력·유지보수 비용 모두 기관 예산 심의를 거쳐야 한다. 이 환경에서 모델 경량화는 다음의 효과를 달성한다.

| 경량화 목표 | 설명 |
|------------|------|
| **하드웨어 요건 완화** | BF16 원본(~20GB VRAM) → AWQ 4-bit(~4.2GB VRAM) → RTX 3060급 GPU에서 운영 가능 |
| **서버 비용 절감** | A100(연간 수천만 원 수준 GPU 서버) 없이 중저가 GPU 서버로 동일 성능 달성 |
| **전력·냉각 비용 절감** | VRAM 사용량 감소 → 전력 소비 감소 → 데이터센터 운영비 절감 |
| **다기관 확장 용이** | 경량 모델은 보급형 서버에도 설치 가능 → 243개 지자체 전체 보급 현실화 |

실측 결과: AWQ 4-bit 양자화 적용 시 모델 크기 **LoRA v2 머지 후 BF16 모델 기준 14.56GB → AWQ 4.94GB (-66.1%)** ([노트북 실측값: `notebooks/M3_issue26/05_merge_and_quantize_v2.ipynb`]). 분류 정확도는 별도 평가 노트북(`notebooks/M3_issue82_evaluation/02_evaluate_standard.ipynb`) 실행 결과를 기준으로 한다.

> AWQ 양자화 기술 근거: [AWQ 논문 - arXiv:2306.00978](https://arxiv.org/abs/2306.00978) (MLSys 2024 Best Paper, MIT Han Lab).

> **⚠ 모델 현황 (2026-03-19)**: 기존 HuggingFace의 AWQ 모델(`umyunsang/civil-complaint-exaone-awq`)은 잘못 학습된 LoRA v1 기반 병합 모델로 생성되어 **폐기** 처리되었다. 현재 재학습된 LoRA v2(`umyunsang/GovOn-EXAONE-LoRA-v2`)를 기반으로 AWQ 양자화 모델을 재생성할 예정이다.

## 7. UI/UX 디자인 및 프론트엔드 요구사항

> **설계 원칙**: 주 고객인 민원 처리 담당 공무원의 업무 흐름에 최적화된 UI를 구현한다. 복잡한 기능 배치보다 단일 화면에서 민원 입력 → AI 분석 → 답변 수정 → 복사 완료의 흐름이 끊기지 않도록 설계한다. 폐쇄망 환경 특성상 외부 CDN, 폰트 서버, 원격 이미지 의존 없이 완전히 오프라인에서 동작해야 한다.

### 7.1 UI 설계 원칙

| 원칙 | 설명 |
|------|------|
| **단순성** | 공무원이 별도 교육 없이 30분 내 사용법 습득 가능 |

### 7.6 프론트엔드 기능 요구사항

> 기존 4.3.1절의 기능 요구사항 중 UI와 직접 관련된 항목을 여기서 상세화한다.

#### FR-UI-001: 민원 입력 및 분석 요청

- 텍스트에어리어는 최소 8줄 이상 표시되며, 최대 3,000자까지 입력 가능하다.
- 입력 내용이 없는 상태에서 분석 버튼 클릭 시 인라인 안내 메시지("민원 내용을 입력해주세요")를 표시한다.
- `Ctrl+Enter` (Windows) / `Cmd+Enter` (Mac) 단축키로 분석을 시작할 수 있다.
- 분석 버튼은 입력 내용이 있을 때만 활성화된다.

#### FR-UI-002: 분류 결과 표시

- 분류 카테고리는 색상 뱃지(도로/교통: 파란색, 환경/위생: 초록색, 복지: 주황색, 세금: 빨간색, 기타: 회색)로 구분 표시한다.
- 신뢰도는 0~100% 수치와 게이지 바를 함께 표시한다.
- 신뢰도 70% 미만인 경우 주의 아이콘(!)과 함께 "담당자 확인 권장" 안내를 표시한다.
- 공무원은 AI 분류 결과를 드롭다운 선택으로 수동 변경할 수 있다.

#### FR-UI-003: 유사 사례 목록 표시

- 유사 사례는 최대 5건을 유사도 점수 내림차순으로 표시한다.
- 각 항목에는 제목(민원 요약), 카테고리, 유사도 점수(%)가 표시된다.
- 항목 클릭 시 PG-002 슬라이드-오버 패널이 열린다.
- 유사 사례가 없는 경우(임계값 미달) "유사 사례를 찾지 못했습니다" 안내 메시지를 표시한다.

#### FR-UI-004: 답변 초안 편집 및 복사

- 생성된 답변 초안은 수정 가능한 텍스트에어리어에 표시된다.
- 텍스트에어리어는 내용에 따라 자동 높이 조절(auto-resize)된다.
- "클립보드 복사" 버튼 클릭 시 현재 텍스트에어리어 내용 전체를 복사하고, 2초간 "복사 완료!" 토스트 알림을 표시한다.
- "초안 재생성" 버튼 클릭 시 확인 다이얼로그 없이 즉시 재요청한다. 기존 초안은 재생성 완료 후 교체된다.
- PG-002의 "이 답변을 초안으로 사용" 클릭 시 해당 유사 사례의 답변 텍스트가 텍스트에어리어에 삽입된다.

#### FR-UI-005: 로딩 및 오류 처리

- AI 처리 중에는 전체 결과 패널에 로딩 스피너를 오버레이하고, 입력 패널의 버튼을 비활성화한다.
- API 응답 시간이 5초를 초과하면 "AI 처리 중입니다. 잠시만 기다려주세요..." 메시지를 추가 표시한다.
- 네트워크 오류 또는 서버 오류(5xx) 발생 시 "분석 중 오류가 발생했습니다. 재시도해주세요." 메시지와 재시도 버튼을 표시한다.
- 재시도 버튼 클릭 시 동일 요청을 재전송한다.

#### FR-UI-006: 시스템 상태 표시

- 상단바 우측에 시스템 상태 뱃지를 항상 표시한다. (정상: 초록, 일부 오류: 주황, 전체 오류: 빨강)
- 뱃지 클릭 시 PG-003 시스템 상태 모달이 열린다.
- 시스템 상태는 30초마다 자동 갱신한다(폴링 방식, 폐쇄망 환경에서 WebSocket 대체).

### 7.8 Figma MCP 연동 방식

> Figma MCP(Model Context Protocol)를 활용하여 디자인-개발 간 전달 오류를 최소화하고 컴포넌트 명세를 코드에 직접 연동한다.

| 항목 | 내용 |
|------|------|
| **디자인 도구** | Figma (팀 공유 파일) |
| **연동 방식** | Figma MCP를 통해 컴포넌트 속성(색상 토큰, 타이포그래피, 스페이싱)을 코드에 직접 참조 |
| **디자인 토큰** | 색상(카테고리별 뱃지 색상, 신뢰도 게이지), 타이포그래피(행정 서체), 스페이싱 시스템을 Figma 변수로 정의 |
| **컴포넌트 매핑** | Figma 컴포넌트 ↔ React 컴포넌트 1:1 매핑 (CMP-001~CMP-010) |
| **배포 제약** | 폐쇄망 환경에서 Figma 접근 불가하므로, 개발 전 디자인 토큰을 `design-tokens.json`으로 추출하여 레포지토리에 포함 |

## 8. 수락 기준

### 8.1 핵심 기능 수락 기준

#### AC-001: 답변 초안 생성
**Given** 민원 본문이 입력되었을 때
**When** "답변 생성" 버튼을 클릭하면
**Then**
- p95 < 3초 이내 답변 초안 생성
- 생성된 답변이 표준 서식(인사말·본문·담당부서·연락처)을 포함
- 답변 길이 200~500자 범위

#### AC-002: 유사 사례 검색
**Given** 민원 본문이 입력되었을 때
**When** 벡터 검색을 실행하면
**Then**
- 상위 5개 유사 민원-답변 쌍 반환
- 검색 시간 p95 < 1초 (FAISS 벡터 DB 레이턴시 기준)
- Recall@5 ≥ 80%

#### AC-003: Docker 폐쇄망 배포
**Given** Docker + nvidia-container-toolkit이 설치된 서버가 준비되었을 때
**When** `docker-compose up -d` 명령을 실행하면
**Then**
- 인터넷 연결 없이 전체 시스템 정상 시작
- `/health` 엔드포인트 200 응답 반환
- 웹 UI Port 3000에서 접근 가능

#### AC-004: 민원 분석 메인 화면 (PG-001)
**Given** 공무원이 행정 내부망에서 브라우저로 시스템에 접속했을 때
**When** 메인 화면이 로딩되면
**Then**
- 초기 페이지 로딩 p95 < 2초 (내부망 기준)
- 좌측 입력 패널과 우측 결과 패널이 1280px 이상 해상도에서 2열 레이아웃으로 표시됨
- 민원 본문 텍스트에어리어가 즉시 포커스되어 붙여넣기 가능한 상태임
- `Ctrl+Enter` (Windows) / `Cmd+Enter` (Mac) 단축키가 동작함

#### AC-005: 분류 결과 수동 선택 기능
**Given** 민원 본문이 입력되고 분석이 완료되었을 때
**When** 분류 결과가 표시되면
**Then**
- 카테고리 뱃지가 정의된 색상 코드로 표시됨
- 공무원이 드롭다운으로 분류를 수동으로 선택·변경할 수 있음
- 자동 분류 신뢰도 수치는 참고용으로만 표시됨 (정확도 요건 없음)

#### AC-006: 유사 사례 표시 및 상세 조회
**Given** 분석 결과가 표시된 상태에서
**When** 유사 사례 목록이 표시되면
**Then**
- 최대 5건이 유사도 내림차순으로 표시됨
- 각 항목에 제목·카테고리·유사도(%)가 표시됨
- 항목 클릭 시 메인 화면을 벗어나지 않고 슬라이드-오버 패널(PG-002)이 열림
- 패널 내 "이 답변을 초안으로 사용" 클릭 시 해당 답변이 초안 텍스트에어리어에 반영됨

#### AC-007: 답변 초안 편집 및 클립보드 복사
**Given** 답변 초안이 생성된 상태에서
**When** 공무원이 초안을 수정하고 "클립보드 복사" 버튼을 클릭하면
**Then**
- 텍스트에어리어 전체 내용이 클립보드에 복사됨
- 2초간 "복사 완료!" 토스트 알림이 화면 우측 하단에 표시됨
- 복사 완료 후 텍스트에어리어 내용이 그대로 유지됨 (삭제되지 않음)

#### AC-008: 오프라인 완전 독립 동작
**Given** 인터넷이 차단된 폐쇄망 환경에서
**When** 시스템을 실행하면
**Then**
- 외부 CDN·폰트 서버·이미지 서버 요청 없이 UI가 정상 렌더링됨
- 브라우저 개발자 도구 네트워크 탭에서 외부 도메인 요청이 0건으로 확인됨
- 모든 정적 자산(폰트·아이콘·JS·CSS)이 로컬 서버에서 서빙됨

#### AC-009: 오류 처리 및 재시도
**Given** AI 서버 응답 실패(네트워크 오류 또는 5xx)가 발생했을 때
**When** 분석 요청이 실패하면
**Then**
- 오류 메시지가 결과 패널에 인라인으로 표시됨
- 재시도 버튼이 표시되며, 클릭 시 동일 요청을 재전송함
- 로딩 인디케이터가 해제되고 입력 패널이 다시 활성화됨

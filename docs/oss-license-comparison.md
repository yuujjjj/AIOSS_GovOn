# OSS 라이선스 비교 및 선택 기준

이 문서는 주요 오픈소스 라이선스를 비교해 `AIOSS_GovOn` 저장소에 적합한 선택 기준을 정리합니다.  
법률 자문을 대체하지 않으며, 실제 조직 정책 반영 전에는 법무 또는 오픈소스 컴플라이언스 검토가 필요합니다.

## 비교 대상

- `MIT`
- `Apache-2.0`
- `GPL-3.0`

위 3개는 실제 현업과 교육 프로젝트에서 가장 자주 비교되는 대표 라이선스 조합입니다.

## 핵심 비교

| 항목 | MIT | Apache-2.0 | GPL-3.0 |
|------|-----|------------|---------|
| 성격 | Permissive | Permissive | Strong Copyleft |
| 사용/수정/재배포 | 매우 자유로움 | 자유로움 | 자유로우나 재배포 조건이 강함 |
| 공지 의무 | 저작권/라이선스 고지 유지 | 라이선스 고지, 변경 표시, `NOTICE` 반영 가능 | 라이선스 유지, 소스 제공 의무가 강함 |
| 특허 조항 | 명시적 특허 허가 없음 | 명시적 특허 허가 포함 | 특허 관련 보호 장치 존재 |
| 폐쇄형 제품 포함 | 용이 | 용이 | 제약 큼 |
| 외부 기여 유도 | 진입장벽 낮음 | 비교적 낮음 | 상대적으로 높음 |
| 조직 내부 공유 | 쉬움 | 쉬움 | 내부 배포 정책 검토 필요 |

## 라이선스별 해석

### MIT

- 장점
  - 조건이 단순해 교육용, 실험용, 스타트업형 프로젝트에 적용이 쉽습니다.
  - 사용, 수정, 병합, 배포, 서브라이선스, 판매까지 폭넓게 허용합니다.
  - 외부 기여자와 내부 사용자 모두 이해하기 쉽습니다.
- 주의점
  - 명시적 특허 허가 조항이 없습니다.
  - 기업 간 공동 개발이나 특허 리스크를 민감하게 보는 조직에서는 다소 약하게 느껴질 수 있습니다.

### Apache-2.0

- 장점
  - Permissive 라이선스이면서도 명시적 특허 허가가 포함됩니다.
  - 변경 파일 표시, 고지 유지, `NOTICE` 관리 등 배포 추적성이 좋습니다.
  - 기업 환경에서 선호되는 경우가 많습니다.
- 주의점
  - MIT보다 의무가 조금 더 많습니다.
  - 작은 교육 프로젝트에는 운영 부담이 다소 늘어날 수 있습니다.

### GPL-3.0

- 장점
  - 파생 저작물과 재배포 시 자유 소프트웨어 성격을 강하게 유지하도록 만듭니다.
  - 소스 제공과 동일 자유 보장을 중요하게 보는 프로젝트에 적합합니다.
- 주의점
  - 폐쇄형 제품이나 혼합 라이선스 환경과의 결합이 어렵습니다.
  - 외부 사용처나 내부 사업부가 재사용을 망설일 수 있습니다.
  - 조직 내 확산보다 “강한 공개 의무”가 우선일 때 더 적합합니다.

## 선택 기준

라이선스 선택은 “무엇을 보호할지”보다 “어떤 협업 구조를 만들고 싶은지”에서 출발하는 편이 실무적으로 더 정확합니다.

### 1. 채택 장벽을 최소화할 것인가

- 예: 수업 프로젝트, PoC, 사내 공통 유틸, 공개 데모
- 권장: `MIT` 또는 `Apache-2.0`

### 2. 특허 관련 문구가 필요한가

- 예: 알고리즘, 플랫폼, SDK, 기업 공동 개발
- 권장: `Apache-2.0`

### 3. 파생물도 반드시 같은 자유를 유지해야 하는가

- 예: 재배포된 수정본도 공개 유지가 핵심 목표
- 권장: `GPL-3.0`

### 4. 외부 연동 라이선스가 다양한가

- 예: 모델, 데이터셋, 서드파티 라이브러리, 사내 코드가 섞임
- 권장: 저장소 자체는 `MIT` 또는 `Apache-2.0`로 단순화하고, 모델/데이터/외부 의존성은 별도 라이선스 표기

### 5. 조직 내부 전파가 중요한가

- 예: 여러 팀이 빠르게 fork, 재사용, 개선해야 함
- 권장: `MIT` 또는 `Apache-2.0`

## AIOSS_GovOn 기준 권고

현재 저장소는 `MIT` 라이선스를 사용하고 있습니다. 이 선택은 아래 이유로 합리적입니다.

- 교육/실습 성격이 강해 기여 장벽이 낮아야 합니다.
- 여러 팀이나 외부 사용자가 빠르게 복제하고 실험하기 쉬워야 합니다.
- `Inner Source` 도입 초기에는 법적 복잡성보다 재사용성과 참여성이 더 중요합니다.
- 저장소 코드와 별도로 모델/데이터 라이선스를 분리 표기하기 쉽습니다.

다만 다음 조건이 생기면 `Apache-2.0` 전환 검토 가치가 있습니다.

- 조직 차원에서 특허 관련 명시 조항을 요구하는 경우
- 배포 시 `NOTICE` 기반의 출처 추적이 필요한 경우
- 향후 외부 협업 주체가 늘어 라이선스 명확성이 더 중요해지는 경우

## 실무 적용 원칙

### 저장소 수준

- 저장소 원본 코드와 문서는 하나의 기본 라이선스로 관리
- 서드파티 모델, 데이터셋, 사전학습 가중치는 별도 라이선스 문서 또는 README 섹션으로 분리 표기

### 배포 수준

- 배포 아카이브에는 `LICENSE` 포함
- Apache-2.0 선택 시 `NOTICE` 운영 여부를 함께 설계
- 모델 카드와 데이터 출처 문서를 함께 유지

### 기여 수준

- `CONTRIBUTING.md`에 “기여 코드는 저장소 라이선스 하에 제출됨”을 명시
- 외부 자산 반입 시 라이선스 검토 체크를 PR 템플릿에 포함

## 최종 판단

현재 단계의 `AIOSS_GovOn`에는 `MIT 유지`가 가장 실용적입니다.

- 이유 1: 기여와 재사용 장벽이 가장 낮음
- 이유 2: 교육용 OSS 실습과 Inner Source 파일럿에 적합
- 이유 3: 조직 확산 초기에는 강한 copyleft보다 참여 확대가 중요

단, 특허 또는 대외 배포 책임이 커지면 `Apache-2.0`이 다음 현실적인 대안입니다.  
`GPL-3.0`은 “재사용 자유”보다 “동일한 공개 의무 유지”를 우선할 때 선택하는 편이 적절합니다.

## 1. Inner Source 개요

### 1.1 Inner Source 정의 및 핵심 원칙

**Inner Source**란 오픈소스 소프트웨어 개발 방법론(투명성, 협업, 코드 리뷰, 커뮤니티 기여)을 **조직 내부**에 적용하는 소프트웨어 개발 전략이다. 소스 코드를 외부에 완전히 공개하지 않으면서도, 조직 내 여러 팀 또는 기관이 하나의 코드베이스에 자유롭게 접근하고 기여할 수 있는 협업 모델을 구축한다.

Inner Source의 핵심 원칙은 다음과 같다.

| 원칙 | 설명 |
|------|------|
| **투명성 (Transparency)** | 소스 코드, 이슈, 의사결정 과정을 조직 내 모든 참여자에게 공개한다. |
| **협업 (Collaboration)** | 팀 또는 기관의 경계를 넘어 누구나 코드에 기여할 수 있다. |
| **멘토링 (Mentoring)** | Trusted Committer가 새로운 기여자를 안내하고 코드 품질을 유지한다. |
| **자발적 참여 (Voluntary Participation)** | 기여는 강제가 아닌 자발적 참여에 기반한다. |
| **문서 중심 (Documentation-first)** | 코드만이 아니라 맥락, 결정 사유, 기여 방법을 문서로 남긴다. |
| **점진적 개방 (Incremental Openness)** | 한 번에 전면 공개가 아닌, 단계적으로 범위를 확대한다. |

### 1.2 오픈소스 vs Inner Source 비교

```
┌─────────────────────────────────────────────────────────────────┐
│                    소프트웨어 개발 스펙트럼                         │
│                                                                 │
│  Closed Source ──── Inner Source ──── Open Source                │
│  (폐쇄형)          (조직내 개방)      (완전 공개)                  │
│                                                                 │
│  ● 단일 팀 개발     ● 조직 내 협업      ● 전 세계 협업             │
│  ● 접근 제한        ● 내부 접근 허용     ● 누구나 접근             │
│  ● 변경 통제        ● 기여 기반 통제     ● 커뮤니티 거버넌스        │
└─────────────────────────────────────────────────────────────────┘
```

| 비교 항목 | 오픈소스 (Open Source) | Inner Source |
|-----------|----------------------|--------------|
| **접근 범위** | 전 세계 누구나 | 조직 내부 (참여 기관) |
| **라이선스** | OSI 인증 라이선스 (MIT, Apache 등) | 내부 라이선스 또는 제한적 공개 라이선스 |
| **기여자** | 외부 불특정 다수 | 내부 승인된 기관/팀 |
| **보안 관리** | 공개 코드에 대한 보안 감사 | 내부 보안 정책 적용, 민감 데이터 분리 용이 |
| **거버넌스** | 커뮤니티 주도 (Foundation 등) | 조직 내 기술 위원회 주도 |
| **속도** | 빠른 혁신, 예측 어려움 | 통제된 혁신, 계획적 릴리즈 |
| **적합 대상** | 범용 도구, 인프라 SW | 기관별 맞춤이 필요한 도메인 SW |

### 1.3 공공기관에서 Inner Source 도입의 필요성

한국의 공공기관 소프트웨어 개발 환경은 다음과 같은 구조적 문제를 안고 있다.

**첫째, 중복 개발의 만연.** 각 지방자치단체와 중앙 부처가 유사한 민원 처리 시스템을 독립적으로 발주하고 개발한다. 민원 분류, 자동 응답, 유사 사례 검색 같은 핵심 기능이 기관마다 별도로 구축되어 예산이 낭비된다.

**둘째, 기술 종속(Vendor Lock-in).** SI 업체에 의존하는 개발 구조로 인해 기관이 자체적으로 시스템을 수정하거나 확장하기 어렵다. 유지보수 비용이 지속적으로 증가하고, 기술 이전이 원활하지 않다.

**셋째, 보안 우려로 인한 폐쇄성.** 민원 데이터의 민감성 때문에 완전한 오픈소스 공개가 어렵다. 그러나 폐쇄적 개발은 코드 품질 저하와 혁신 정체를 초래한다.

**Inner Source는 이 세 가지 문제를 동시에 해결한다.** 코드를 기관 간에 공유하여 중복 개발을 방지하고, 기관 자체 역량으로 커스터마이징이 가능하며, 외부 공개 없이 내부에서만 협업하므로 보안 요건을 충족한다.

```
┌──────────────────────────────────────────────────────────────┐
│                공공기관 SW 개발의 구조적 문제                     │
│                                                              │
│   기관 A ──┐                                                 │
│   기관 B ──┤── 유사 시스템 독립 개발 ── 예산 낭비               │
│   기관 C ──┘                                                 │
│                                                              │
│                    ▼ Inner Source 도입 후 ▼                    │
│                                                              │
│   기관 A ──┐                                                 │
│   기관 B ──┤── 공통 코어 공유 + 기관별 커스터마이징              │
│   기관 C ──┘        ↕ 기여/피드백 순환                         │
│                                                              │
│   결과: 비용 절감 + 품질 향상 + 보안 유지                       │
└──────────────────────────────────────────────────────────────┘
```

특히 GovOn과 같은 **AI 기반 민원 처리 시스템**은 Inner Source에 최적의 대상이다.

- **모델 파인튜닝 데이터**는 기관별로 다르지만, 학습 파이프라인은 공유 가능하다.
- **RAG 검색 인덱스**는 기관별 법령/매뉴얼로 교체하면 되므로 엔진 자체는 재사용 가능하다.
- **API 인터페이스**는 표준화하면 프론트엔드를 기관별로 자유롭게 구성할 수 있다.

## 2. GovOn Inner Source 도입 전략

### 2.1 현재 프로젝트 구조의 Inner Source 준비도 분석

GovOn의 현재 아키텍처를 Inner Source 준비도 관점에서 평가한다.

```
src/
├── data_collection_preprocessing/   # 데이터 수집 파이프라인
├── training/                        # QLoRA 파인튜닝
├── quantization/                    # AWQ 양자화
├── inference/                       # FastAPI 서빙 (핵심)
│   ├── api_server.py               # vLLM 엔진 관리 + 엔드포인트
│   ├── retriever.py                # FAISS 검색 (multilingual-e5-large)
│   ├── index_manager.py            # 다중 인덱스 관리
│   ├── hybrid_search.py            # BM25 + Dense 하이브리드 검색
│   ├── document_processor.py       # 문서 처리
│   ├── schemas.py                  # Pydantic 스키마
│   ├── vllm_stabilizer.py          # EXAONE 런타임 패치
│   └── db/                         # SQLAlchemy ORM, Alembic
├── evaluation/                     # 모델 평가
└── utils/                          # 공통 유틸리티
```

#### 2.2.1 민원 분류 모델 - 도메인별 파인튜닝 분리

현재 GovOn은 EXAONE-Deep-7.8B를 QLoRA로 파인튜닝하여 민원을 분석한다. 타 기관 적용을 위해 다음과 같이 분리한다.

```
training/
├── core/                          # [공유] 공통 학습 인프라
│   ├── trainer.py                 # SFTTrainer 래퍼
│   ├── data_loader.py             # 범용 데이터 로더
│   ├── evaluation.py              # 평가 메트릭
│   └── config_schema.py           # 학습 설정 스키마
├── adapters/                      # [공유] 도메인 어댑터
│   ├── base_adapter.py            # 추상 어댑터 클래스
│   ├── civil_complaint.py         # 민원 처리 도메인
│   ├── welfare.py                 # 복지 상담 도메인 (확장 예시)
│   └── tax.py                     # 세무 상담 도메인 (확장 예시)
└── configs/                       # [기관별] 기관 맞춤 설정
    ├── default.yaml               # 기본 학습 설정
    └── agency_template.yaml       # 기관 설정 템플릿
```

**핵심 설계 원칙:**
- **Base Model 공유**: EXAONE-Deep-7.8B 기반 모델은 모든 기관이 동일하게 사용한다.
- **LoRA 어댑터 분리**: 기관별 파인튜닝 결과를 독립적인 LoRA 어댑터로 관리한다.
- **학습 파이프라인 표준화**: `trainer.py`의 인터페이스를 고정하고, 데이터 형식만 변환한다.
- **설정 기반 커스터마이징**: YAML 설정 파일로 하이퍼파라미터, 데이터 경로, 도메인 특화 옵션을 관리한다.

#### 2.2.2 RAG 검색 엔진 - 인덱스 교체 가능 설계

현재 GovOn의 `MultiIndexManager`는 CASE, LAW, MANUAL, NOTICE 네 가지 인덱스 유형을 지원한다. 이를 확장 가능한 플러그인 구조로 발전시킨다.

```python
# 인덱스 교체 인터페이스 (설계 방향)
class IndexProvider(ABC):
    """기관별 인덱스를 제공하는 추상 인터페이스"""

    @abstractmethod
    def get_index_config(self) -> dict:
        """인덱스 유형, 경로, 임베딩 설정을 반환"""
        pass

    @abstractmethod
    def build_index(self, documents: list) -> None:
        """문서 목록으로부터 FAISS 인덱스를 구축"""
        pass

    @abstractmethod
    def get_search_weights(self) -> dict:
        """하이브리드 검색 가중치를 반환"""
        pass
```

```
inference/
├── search/                        # [공유] 검색 엔진 코어
│   ├── retriever.py               # FAISS 기반 Dense 검색
│   ├── bm25_indexer.py            # BM25 희소 검색
│   ├── hybrid_search.py           # 하이브리드 검색 통합
│   └── search_cache.py            # 검색 결과 캐싱
├── index_providers/               # [기관별] 인덱스 제공자
│   ├── base_provider.py           # 추상 인덱스 제공자
│   ├── civil_complaint.py         # 민원 인덱스 (기본)
│   └── README.md                  # 커스텀 인덱스 작성 가이드
└── embeddings/                    # [공유] 임베딩 모델 관리
    ├── model_registry.py          # 임베딩 모델 레지스트리
    └── e5_embedder.py             # multilingual-e5-large 래퍼
```

**교체 가능 설계의 핵심:**
- **인덱스 유형 확장**: 기관이 자체 문서 유형(조례, 내부 규정, FAQ 등)을 인덱스로 추가할 수 있다.
- **임베딩 모델 선택**: 기관 요구에 따라 다른 임베딩 모델을 사용할 수 있도록 레지스트리 패턴을 적용한다.
- **하이브리드 검색 가중치 조정**: 기관별 문서 특성에 따라 Dense/Sparse 검색 비율을 조정할 수 있다.

#### 2.2.3 API 서버 - 범용 민원 처리 인터페이스

API 서버는 기관에 관계없이 동일한 인터페이스를 제공해야 한다. 현재 GovOn의 FastAPI 기반 서버를 범용화한다.

```
API 계층 구조:

/api/v1/
├── /complaints                    # 민원 처리 (공통)
│   ├── POST /analyze              # 민원 분석 요청
│   ├── POST /classify             # 민원 분류
│   └── GET  /status/{id}          # 처리 상태 조회
├── /search                        # 검색 (공통)
│   ├── POST /similar              # 유사 민원 검색
│   └── POST /documents            # 관련 문서 검색
├── /health                        # 헬스체크 (공통)
│   ├── GET  /ready                # 서비스 준비 상태
│   └── GET  /live                 # 서비스 생존 상태
└── /admin                         # 관리 (기관별 확장)
    ├── GET  /metrics              # 사용 통계
    └── POST /index/rebuild        # 인덱스 재구축
```

**API 표준화 원칙:**
- **버전 관리**: URL 기반 버전 관리(`/api/v1/`)로 하위 호환성을 보장한다.
- **스키마 계약**: Pydantic 모델로 요청/응답 스키마를 엄격히 정의한다.
- **인증 추상화**: API Key 기반 인증을 기본 제공하되, 기관별 SSO/SAML 연동이 가능하도록 인증 미들웨어를 플러그인화한다.
- **에러 코드 표준화**: 기관 간 일관된 에러 코드 체계를 사용한다.

### 2.3 기여 모델: 역할 정의

Inner Source의 핵심은 명확한 역할 정의와 기여 프로세스이다. InnerSource Commons에서 제시하는 역할 모델을 GovOn에 맞게 적용한다.

```
│  ┌────────────────────▼────────────────────────────────┐    │
```

### Phase 2: 파일럿 기관 도입 (2026 Q4 ~ 2027 Q1)

**목표**: 1~2개 공공기관에 GovOn을 도입하고, Inner Source 기여 프로세스를 검증한다.

- **광역 지자체 민원실**: 다양한 민원 유형, 충분한 데이터 보유

### Phase 3: 확산 및 커뮤니티 구축 (2027 Q2 ~ Q3)

**목표**: 파일럿 성과를 바탕으로 참여 기관을 확대하고, 자생적 커뮤니티를 형성한다.

#### 3-1. 기여 가이드 고도화

**CONTRIBUTING.md 핵심 내용:**

```markdown
## 기여 유형
- **버그 수정**: 이슈 태그 `bug`가 붙은 항목
- **기능 추가**: 이슈 태그 `enhancement`, 사전 논의 필수
- **문서 개선**: 오타 수정부터 가이드 작성까지
- **기관 확장 모듈**: 새로운 도메인 어댑터, 인덱스 제공자

## 브랜치 전략
- main: 안정 릴리즈 브랜치, 직접 push 금지
- feat/이슈번호-설명: 기능 개발
- fix/이슈번호-설명: 버그 수정

## PR 규칙
- 모든 PR은 최소 1명의 TC 리뷰 필수
- CI 파이프라인 통과 필수 (테스트, 린트, 타입체크)
- 커밋 메시지는 Conventional Commits 형식 (한글)
```

#### 3-2. 이슈 트래커 운영 체계

| 라벨 | 설명 | 대응 SLA |
|------|------|---------|
| `bug` | 버그 보고 | TC 48시간 내 분류 |
| `enhancement` | 기능 요청 | 주간 회의에서 논의 |
| `good-first-issue` | 초보 기여자용 | 상시 5건 이상 유지 |
| `agency-specific` | 기관 특화 요구 | 해당 기관 TC 대응 |
| `security` | 보안 이슈 | 24시간 내 대응 |
| `documentation` | 문서 관련 | TC 72시간 내 분류 |

#### 3-3. 커뮤니티 활동 체계

| 활동 | 주기 | 참여자 | 형식 |
|------|------|--------|------|
| **데일리 스탠드업** | 매일 | 코어 개발팀 | 비동기 (Slack/Teams) |
| **TC 위클리 싱크** | 주 1회 | 전체 TC | 화상 회의 30분 |
| **커뮤니티 먼슬리** | 월 1회 | 전체 기여자 | 화상 회의 1시간 |
| **기술 세미나** | 분기 1회 | 전체 참여자 | 온/오프라인 병행 |
| **해커톤** | 반기 1회 | 전체 참여자 | 오프라인 1일 |

### Phase 4: 거버넌스 체계 확립 (2027 Q4 이후)

**목표**: 지속 가능한 Inner Source 생태계를 위한 제도적 기반을 구축한다.

#### 4-1. 기술 위원회 (Technical Steering Committee) 구성

```
┌─────────────────────────────────────────────────────┐
│              기술 위원회 (TSC)                         │
│                                                     │
│  구성: 5~7명                                         │
│  - GovOn 원 개발팀 대표 (2명)                         │
│  - 파일럿 기관 기술 대표 (2명)                         │
│  - 확산 기관 대표 (1~2명)                             │
│  - 외부 자문위원 (1명, AI/공공 SW 전문가)              │
│                                                     │
│  회의: 분기 1회 정기 + 필요 시 임시                    │
│                                                     │

│  - 새로운 TC 임명 및 역할 변경                         │
│  - 라이선스 및 보안 정책 변경                          │

└─────────────────────────────────────────────────────┘
```

#### 4-2. 라이선스 관리

**라이선스 선택 시 고려 사항:**

| 고려 사항 | 설명 |
|-----------|------|
| **공공 SW 특성** | 정부 예산으로 개발된 SW의 공공성 확보 |
| **기관 간 공유** | 참여 기관 간 자유로운 사용/수정 허용 |
| **외부 공개 제한** | 민감 데이터 처리 로직의 외부 유출 방지 |
| **상용 SW 연동** | 기관 기존 시스템과의 통합 허용 |
| **특허 보호** | AI 모델 관련 특허 이슈 대응 |

**라이선스 후보 비교:**

| 라이선스 | 공유 범위 | Copyleft | 특허 조항 | 적합도 |
|---------|----------|----------|----------|--------|
| EUPL 1.2 | 유럽 공공기관용 | Weak | 있음 | ★★★★☆ |
| 한국 공공 SW 라이선스 (검토 중) | 국내 공공기관 | 유연 | 검토 필요 | ★★★★★ |
| Apache 2.0 (내부 한정) | 참여 기관 | 없음 | 있음 | ★★★☆☆ |
| 자체 Inner Source 라이선스 | 참여 기관 | 커스텀 | 커스텀 | ★★★★☆ |

**권장안**: EUPL 1.2를 기반으로 국내 공공기관 특성에 맞는 부가 조항을 추가한 **커스텀 Inner Source 라이선스**를 채택한다. 향후 한국 공공 SW 라이선스 표준이 수립되면 마이그레이션한다.

## 4. 예상 효과 및 리스크

### 4.1 기대 효과

#### 정량적 효과

| 효과 항목 | 추정치 | 산출 근거 |
|-----------|--------|----------|
| **개발 비용 절감** | 기관당 40~60% | 공통 코어 재사용으로 신규 개발 범위 축소 |
| **도입 기간 단축** | 12개월 → 4개월 | 검증된 코어 + 기관 커스터마이징만 수행 |
| **코드 품질 향상** | 결함 밀도 30% 감소 | 다수 기관의 코드 리뷰 및 테스트 기여 |
| **유지보수 비용 절감** | 연간 25~35% | 공통 패치 공유, 중복 수정 방지 |

#### 정성적 효과

**기술 역량 내재화**: 기관 자체 개발 역량이 강화되어 SI 의존도가 줄어든다. 기여 과정에서 개발자의 AI/ML 역량이 자연스럽게 성장한다.

**코드 품질의 선순환**: 여러 기관의 개발자가 코드를 리뷰하고 개선하면서, 단일 팀 개발에서는 발견하기 어려운 설계 결함이나 성능 이슈가 조기에 발견된다.

**정책 표준화 촉진**: 민원 처리의 핵심 로직을 공유함으로써 기관 간 민원 처리 품질의 편차가 줄어든다. 시민이 어느 기관에 민원을 넣어도 일관된 수준의 AI 지원을 받을 수 있다.

**공공 AI 생태계 활성화**: GovOn Inner Source가 성공하면, 다른 공공 AI 프로젝트(챗봇, 문서 요약, 감정 분석 등)에도 동일한 모델을 적용할 수 있는 선례가 된다.

- **기관별 TC 할당**: 각 참여 기관에서 최소 1명의 TC를 배출하도록 권장

- **기여자 인센티브**: 우수 기여자 표창, 기관 내 승진 평가 반영 권고

- **RFC(Request for Comments) 프로세스**: 주요 변경 사항은 RFC 문서를 작성하고 1주일 이상 의견 수렴 후 결정
- **투표 규칙**: TSC 의결은 참석 인원 2/3 이상 찬성으로 확정
- **갈등 해소 메커니즘**: 기관 간 의견 충돌 시 TSC 의장이 중재, 필요 시 외부 자문위원 자문

## 5. 참고 사례

### 5.1 기업 Inner Source 성공 사례

#### Microsoft

Microsoft는 Inner Source의 대표적 성공 사례이다. 2010년대 중반부터 사내 전체 소스 코드를 내부 개방하기 시작했으며, **1ES(One Engineering System)**를 통해 전사 코드베이스를 통합 관리한다.

| 항목 | 내용 |
|------|------|
| **규모** | 10만+ 개발자, 수천 개 내부 리포지토리 |
| **핵심 성과** | Windows, Office 등 핵심 제품 간 코드 공유 활성화 |
| **교훈** | 경영진의 문화적 지원이 기술적 도구보다 중요 |
| **GovOn 적용점** | 기관장 레벨의 지원 확보, 문화 변화를 기술 도입과 병행 |

#### SAP

SAP는 2016년부터 체계적인 Inner Source 프로그램을 운영하며, InnerSource Commons 재단의 설립 멤버이다.

| 항목 | 내용 |
|------|------|
| **규모** | 2만+ 개발자, 1,000+ Inner Source 프로젝트 |
| **핵심 성과** | 제품 간 중복 코드 50% 감소, 코드 품질 지표 향상 |
| **운영 모델** | Inner Source Program Office(ISPO) 전담 조직 운영 |
| **GovOn 적용점** | 전담 조직(ISPO 상당) 설치, 체계적 메트릭 수집 |

#### PayPal

PayPal은 Inner Source를 통해 사일로화된 팀 구조를 혁신한 사례이다. "Inner Source를 하지 않을 이유를 찾지 못했다"는 Cedric Williams(전 PayPal 엔지니어)의 발언으로 유명하다.

| 항목 | 내용 |
|------|------|
| **규모** | 수천 명 개발자, 주요 결제 시스템 |
| **핵심 성과** | 팀 간 코드 기여 300% 증가, 배포 주기 단축 |
| **특이점** | 하향식(Top-down)이 아닌 풀뿌리(Bottom-up) 도입 |
| **GovOn 적용점** | 파일럿 기관에서 성공 후 상향 확산하는 전략 참고 |

#### Europace (금융 플랫폼)

Europace는 독일 부동산 금융 플랫폼으로, **규제 산업에서 Inner Source를 성공적으로 도입한 사례**이다. 민감한 금융 데이터를 다루면서도 Inner Source를 운영한 경험은 GovOn에 직접적으로 참고할 수 있다.

| 항목 | 내용 |
|------|------|
| **규모** | 수백 명 개발자, 금융 규제 준수 |
| **핵심 성과** | 규제 환경에서도 Inner Source 운영 가능함을 입증 |
| **특이점** | 데이터 보안과 코드 공유를 양립하는 거버넌스 모델 |
| **GovOn 적용점** | 민감 데이터(민원) 처리 환경에서의 Inner Source 거버넌스 참고 |

유럽연합은 **European Union Public Licence (EUPL)**를 제정하여 회원국 간 공공 소프트웨어 공유를 제도적으로 지원한다. Joinup 플랫폼(joinup.ec.europa.eu)을 통해 공공기관 간 소프트웨어를 공유하며, 이는 Inner Source의 국가 간 확장 모델이라 볼 수 있다.

미국 연방정부는 **code.gov** 정책을 통해 연방 기관이 개발한 코드의 20% 이상을 오픈소스로 공개하도록 권장한다. GSA 산하 **18F** 팀은 공공 디지털 서비스를 오픈소스로 개발하는 선도 조직이다.

## 6. 결론

### 6.1 Inner Source 도입의 전략적 의의

GovOn은 온디바이스 AI 민원 분석 처리 시스템으로서, 그 기술적 구조(모듈화된 파이프라인, 표준화된 API, 교체 가능한 RAG 인덱스)가 Inner Source에 높은 적합성을 보인다. Inner Source 도입을 통해 단일 프로젝트에서 **공공기관 AI 민원 처리의 공통 플랫폼**으로 발전할 수 있다.

### 6.2 핵심 성공 요인

1. **경영진/기관장 지원 확보**: 기술적 도입보다 문화적 변화가 선행되어야 한다. 기관장 레벨에서 Inner Source의 가치를 이해하고 인력/예산을 지원해야 한다.

2. **점진적 도입**: Phase 1~4를 급하게 압축하지 않고, 각 단계의 성과를 검증한 후 다음 단계로 진행한다. 특히 Phase 2 파일럿의 성공이 전체 로드맵의 핵심 분기점이다.

3. **문서 우선 문화**: "코드만 공유하는 것"이 아니라 "맥락과 결정 사유까지 공유하는 것"이 Inner Source의 핵심이다. 문서화에 대한 투자를 기술 개발과 동등하게 취급해야 한다.

4. **보안과 개방의 균형**: 민원 데이터의 민감성을 존중하면서도 코드 공유의 이점을 극대화하는 거버넌스 모델이 필요하다. 코드와 데이터의 완전 분리가 이 균형의 핵심이다.

5. **커뮤니티 자생력 확보**: 장기적으로 원 개발팀의 주도 없이도 커뮤니티가 자체적으로 운영될 수 있어야 한다. TC 교대 운영, 기관별 TC 배출, 기여자 인센티브가 이를 가능하게 한다.

### 6.3 향후 전망

Inner Source가 성공적으로 정착되면, GovOn은 궁극적으로 **제한적 오픈소스(Restricted Open Source)** 또는 **완전 오픈소스**로의 전환도 고려할 수 있다. 이는 한국 공공 AI 생태계의 성숙도와 보안 환경 변화에 따라 결정될 사안이다.

```
Inner Source 성숙도 단계:

Phase 1~2        Phase 3~4          향후
[내부 공유] ──▶ [기관 간 공유] ──▶ [제한적 공개] ──▶ [오픈소스]
단일 팀          참여 기관           공공기관 전체      전 세계
```

GovOn Inner Source의 성공은 단순히 하나의 민원 처리 시스템을 넘어서, **한국 공공기관의 소프트웨어 개발 문화를 혁신하는 시금석**이 될 수 있다. 기관 간 벽을 허물고, 시민을 위한 더 나은 디지털 서비스를 함께 만들어가는 것이 Inner Source의 궁극적 목표이다.
